# Schema Contracts And Versioning

## Why this exists

Coyote3 uses domain-oriented Pydantic DB contracts for collection writes. Admin/resource creation is validated by backend contracts before DB writes.

Managed-resource wiring lives in one registry:

- `api/contracts/managed_resources.py`

## Contract package layout

DB contracts are organized in `api/contracts/schemas/`:

- `base.py`: shared permissive base (`extra="allow"`) and version-history entry model.
- `samples.py`: sample document model and DNA/RNA key-consistency rules.
- `dna.py`: variants, CNVs, translocations, biomarkers, coverage-related models.
- `rna.py`: fusions, RNA expression/classification/QC models.
- `assay.py`: ASP/ASPC/ISGL and assay-linked blacklist models.
- `governance.py`: users, roles, permissions, schema-definition documents.
- `reference.py`: external/reference and annotation contracts.
- `registry.py`: collection-to-model adapter map and validation helpers.

## Single source mapping

`api/contracts/managed_resources.py` is the single source that links:

- admin resource key (for example `role`, `user`, `asp`, `aspc_dna`)
- target Mongo collection contract (for example `roles`, `users`, `asp_configs`)
- contract-owned UI schema identity (`schema_type`, `schema_category`, `schema_id`)

Managed resources currently covered end-to-end:

- ASP (`assay_specific_panels`)
- ASPC DNA (`asp_configs`)
- ASPC RNA (`asp_configs`)
- ISGL (`insilico_genelists`)
- Users (`users`)
- Roles (`roles`)
- Permissions (`permissions`)

## Write-path enforcement

Create/update flows validate and normalize payloads against contracts before DB writes:

- Internal ingest collection endpoints (`/api/v1/internal/ingest/collection*`)
- Admin create/update for:
  - ASP (`assay_specific_panels`)
  - ASPC (`asp_configs`)
  - ISGL (`insilico_genelists`)
  - Users (`users`)
  - Roles (`roles`)
  - Permissions (`permissions`)

This gives strict key/type checks while still allowing forward-compatible extra keys.

## UI schema source

For managed admin resources (`asp`, `aspc`, `isgl`, `users`, `roles`, `permissions`),
UI form schema payloads are generated by backend code from Pydantic contracts and
managed-resource mapping.

- Mapping: `api/contracts/managed_resources.py`
- Generated UI schema: `api/contracts/managed_ui_schemas.py`

UI form definitions for these resources are backend-owned and generated from contracts.

Generated payloads include UI-ready structure:

- `fields` with `data_type`, `display_type`, `required`, `default`, and `options`
- ordered `sections` used directly by admin templates
- system metadata fields (`created_by`, `created_on`, `updated_by`, `updated_on`, `version`)
- resource-specific field behavior (for example gene upload/edit fields and JSON editors)

Operational flow:

1. UI requests `create_context` from API.
2. API builds schema from managed contracts.
3. UI renders fields from returned `sections` + `fields`.
4. UI submits form payload only.
5. API services stamp `version` and `version_history`.
6. API validates payload with collection contract and writes to Mongo.

This keeps UI schema behavior and DB-write validation in one backend source of truth.

## DB `schemas` collection status

Managed admin forms use backend-owned definitions in `api/contracts/managed_ui_schemas.py`.
Collection validation is enforced by Pydantic contracts in `api/contracts/schemas/*`.

## Versioning and stability

- Keep `version` increments on updates.
- Keep `version_history` entries on create/update mutations.
- Prefer additive contract changes and explicit migration plans for breaking changes.
- Use strict contracts where key-level lock is required (`_StrictDocBase`).

## Recommended evolution pattern

1. Add/adjust Pydantic fields in `api/contracts/schemas/*`.
2. If form behavior changes, update resource UI metadata in `api/contracts/managed_ui_schemas.py` (sections/display types/overrides).
3. Bump managed contract version in `api/contracts/managed_resources.py` when version semantics change.
4. Update seed fixtures with new keys (`tests/fixtures/db_dummy/*`) when baseline data changes.
5. Update docs generated from contracts (`scripts/export_collection_contracts_doc.py`) and relevant developer/product pages.
6. Add/update tests in `tests/unit/test_db_documents.py` and admin-service tests.
7. Run `bash scripts/check_contract_integrity.sh` before merge.
7. If a breaking key change is unavoidable, add migration script and runbook entry.
